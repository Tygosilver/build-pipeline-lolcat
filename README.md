# Локальный конвейер сборки проекта lolcat

## Описание

Автоматизированный локальный конвейер сборки для проекта **lolcat** — утилиты командной строки, преобразующей текстовый вывод в радужные цвета. Конвейер работает в изолированной среде Docker и поддерживает три режима сборки: **Release**, **Debug** и **Coverage** с проверкой регрессии покрытия кода.

Архитектура конвейера универсальна и может быть адаптирована под любой C/C++ проект с минимальными изменениями.

## Требования

- **Docker** (версия 20.10+) — для изоляции среды сборки
- **Bash** — для выполнения скриптов оркестрации (предустановлен в большинстве Linux-дистрибутивов)

## Как запустить

Конвейер запускается одной командой с указанием режима сборки:

```bash
./build.sh --mode=<release|debug|coverage>
```

### Примеры запуска

```bash
# Сборка в режиме Release (оптимизированная для конечных пользователей)
./build.sh --mode=release

# Сборка в режиме Debug (с отладочными символами)
./build.sh --mode=debug

# Сборка в режиме Coverage (с измерением покрытия кода тестами)
./build.sh --mode=coverage
```

**Первый запуск:** при отсутствии Docker-образа конвейер автоматически соберёт образ из Dockerfile (займёт 1-2 минуты).
**Последующие запуски:** используется существующий образ — сборка занимает 5-10 секунд благодаря кэшированию через ccache.

## Режимы сборки

**Release**

- Оптимизация скорости (-O2 -DNDEBUG)
- Удаление отладочной информации (strip)
- Генерация .deb пакета с уникальным именем: lolcat-release_1.0-rev<номер>_build_<временная_метка>_amd64.deb
- Предназначен для конечных пользователей

**Debug**

- Отладочные символы (-g) без оптимизации (-O0)
- Сохранение отладочной информации (без strip)
- Генерация .deb пакета с уникальным именем: lolcat-debug_1.0-rev<номер>_build_<временная_метка>_amd64.deb
- Предназначен для разработчиков (работа с отладчиками: gdb, lldb)

**Coverage**

- Инструментирование кода для измерения покрытия (--coverage)
- Выполнение интеграционного теста (обработка тестовых данных через утилиту)
- Генерация .deb пакета и HTML-отчёта о покрытии кода
- Ключевая особенность: автоматическая проверка регрессии покрытия:
    - При уменьшении покрытия по сравнению с предыдущим запуском — сборка завершается с ошибкой
    - Сообщение: РЕГРЕССИЯ ПОКРЫТИЯ: X% < Y%
    - Это гарантирует, что новые изменения не ухудшают тестовое покрытие

## Структура проекта

```
build-pipeline/
├── build.sh                # Главный скрипт оркестрации конвейера
├── Dockerfile              # Рецепт сборки образа среды разработки
├── README.md               # Документация
├── scripts/
│   ├── build_release.sh    # Сборка в режиме Release
│   ├── build_debug.sh      # Сборка в режиме Debug
│   └── build_coverage.sh   # Сборка в режиме Coverage с проверкой регрессии
├── src/
│   └── lolcat/             # Исходный код проекта lolcat (клонирован из GitHub)
├── artifacts/              # Артефакты сборки (пакеты .deb)
│   ├── lolcat-release_*.deb
│   ├── lolcat-debug_*.deb
│   └── lolcat-coverage_*.deb
└── reports/                # Отчёты о сборках
    ├── build_report_*.txt  # История сборок с метаданными
    ├── last_coverage.txt   # Текущее значение покрытия (для сравнения)
    └── coverage_build_*/   # HTML-отчёты о покрытии кода (открываются в браузере)
```

## Артефакты

### Release и Debug

- Собранный бинарник lolcat в пакете .deb
- Имя пакета: lolcat-<режим>_1.0-rev<номер>_<временная_метка>_amd64.deb
- Имя пакета содержит номер ревизии из истории Git и уникальную временную метку

### Coverage

- Собранный бинарник lolcat с инструментированием в пакете .deb
- HTML-отчёт о покрытии кода: reports/coverage_<временная_метка>/index.html
- Значение покрытия сохраняется в reports/last_coverage.txt для сравнения

## Отчёты о сборке

После каждого запуска создаётся файл в формате:

```
Номер запуска: 3
Уникальный номер ревизии: 93
Тип сборки: coverage
Значение покрытия: 19.5
Дата и время: 2026-02-03 23:19:09
Артефакты: artifacts/lolcat_*-rev93.*
```

Файлы отчётов имеют уникальные имена: build_report_YYYYMMDD_HHMMSS.txt

## Примечания об упрощениях

1. **Выбор проекта для сборки:**
    - Использован минималистичный проект lolcat (один файл на C, ~300 строк)
    - без сложных зависимостей и системы сборки (Makefile/CMake)
    - Упрощает демонстрацию концепции конвейера без отвлечения на сложность проекта
2. **Проверка регрессии покрытия:**
    - Простое сравнение с предыдущим запуском через файл last_coverage.txt
    - Без хранения полной истории покрытия или интеграции с системой контроля версий
    - Достаточно для демонстрации требования «сборка падает при уменьшении покрытия»
3. **Интеграционное тестирование:**
    - Простой тест — обработка одной строки текста через утилиту
    - Без полноценного набора тестов или фреймворка тестирования
    - Демонстрирует принцип работы инструментов покрытия кода
4. **Упаковка в пакеты:**
    - Только DEB-пакеты (без RPM), как указано в требованиях
    - Минимальная структура пакета через утилиту fpm
    - Базовые метаданные (имя, версия, архитектура, описание)
5. **Управление артефактами:**
    - Все артефакты сохраняются в одну директорию artifacts/
    - Уникальные имена файлов через временную метку вместо разделения по поддиректориям
    - Упрощает навигацию и проверку результатов
6. **Обработка ошибок:**
    - Базовые проверки существования файлов после критических операций (компиляция, strip)
    - Понятные сообщения об ошибках на русском языке
    - Без избыточной валидации входных данных (предполагается корректный запуск)
7. **Архитектура скриптов:**
    - Прямолинейная структура без избыточных абстракций
    - Каждый режим сборки — отдельный скрипт для наглядности
    - Комментарии на русском языке для понимания логики каждым шагом
8. **Кэширование:**
    - Интеграция ccache на уровне образа (без настройки постоянного хранилища)
    - Ускорение повторных сборок в рамках одной сессии
    - Без сложной настройки постоянного кэша между перезапусками системы

## Автоматизация

- Docker образ собирается автоматически при первом запуске
- Проверка наличия образа перед каждой сборкой
- Номер ревизии определяется автоматически через git rev-list --count HEAD
- Временная метка генерируется автоматически для уникальности артефактов
- Отчёты генерируются автоматически после каждой сборки
- Права доступа к артефактам исправляются автоматически через chown
